## 术语说明

### 1. 交易流水
指用户从银行、支付宝、微信支付等平台导出的、能够反映收入或支出的原始账单记录。

### 2. 标准化交易记录
指系统完成导入解析后，统一映射为内部结构的数据记录，包含交易时间、金额、收支方向、商户、备注、分类等字段。

### 3. 自动分类
指系统根据内置语义规则或用户补充规则，对交易流水自动归入某个消费类别的过程。

### 4. 人工修正
指用户对自动分类结果进行手动修改，使单条交易的最终分类更准确。

### 5. 分类规则
指用于判断交易应归属哪个分类的简单匹配逻辑，V1 主要基于商户名、关键词、备注或摘要文本进行匹配。

### 6. 未分类
指未被任何默认规则或用户规则命中的交易状态。该状态在 V1 中是允许存在的中间结果，不视为系统错误。

### 7. 月总预算
指用户针对某一自然月设置的总支出上限。

### 8. 分类预算
指用户针对某一消费分类设置的月度支出上限。

### 9. 预算预警
指系统根据预算使用率给出的状态提示。V1 中包括：
- 正常
- 接近预算
- 已超支

### 10. 看板
指首页集中展示消费总览、分类分布、趋势分析、预算状态和提醒信息的综合页面。

## 一、产品概述

### 1.1 项目名称
个人财务智能语义看板（Smart Finance Dashboard）

### 1.2 产品定位
个人财务智能语义看板是一款面向普通个人用户和家庭用户的 Web 响应式财务管理产品。产品核心目标不是替代专业财务系统，而是帮助用户以更低成本完成账单整理、消费分类、支出洞察和预算控制。

用户通过导入银行或支付平台导出的账单文件，系统自动解析交易流水，并基于语义规则完成消费分类，进一步生成消费结构分析、预算执行情况和超支预警信息，帮助用户更直观地理解自己的消费情况。

### 1.3 目标用户

1. 有日常收支管理需求的个人用户
这类用户通常通过银行卡、支付宝、微信支付等方式完成日常消费，希望定期整理账单，但不愿长期坚持手工记账。

2. 有家庭消费统筹需求的普通家庭用户
这类用户更关注每月整体支出水平、主要消费类别和预算执行情况，希望借助可视化方式了解家庭消费结构。

### 1.4 核心价值

1. 自动化分类，减少人工记账成本
用户通过上传账单文件即可获得初步分类结果，系统通过默认语义规则识别餐饮、交通、购物等常见消费类型，并允许用户补充自己的规则。

2. 可视化洞察，帮助用户理解消费结构
系统以看板形式展示支出总额、分类占比、时间趋势和预算状态，让用户不必逐条查看流水即可形成整体认知。

3. 预算管理与预警，辅助理性消费
用户可以设置月总预算和分类预算，系统根据实际支出进度提供预算执行状态和超支提醒。

## 二、产品目标与范围

### 2.1 产品目标

1. 用户能够上传账单文件，并成功获得结构化交易记录。
2. 用户能够获得基础可用的自动分类结果，减少人工整理负担。
3. 用户能够通过可视化看板快速理解月度消费结构。
4. 用户能够设置预算并获得明确的预算预警。
5. 整体体验足够简单，让普通用户愿意持续使用。

### 2.2 第一版范围（In Scope）

1. 单用户使用
2. Web 响应式产品形态
3. 账单文件上传与解析
4. 交易流水自动分类
5. 手动修正分类
6. 用户补充简单分类规则
7. 消费分析看板
8. 月总预算与分类预算
9. 预算超支预警

### 2.3 非目标（Out of Scope）

1. 借贷、分期、信用卡还款计划等复杂负债管理
2. 股票、基金、保险、理财产品等资产管理
3. 多用户协作、家庭账本共享、权限体系
4. 与银行、支付宝、微信等平台 API 的实时直连
5. 原生 iOS / Android 应用
6. 复杂规则引擎
7. AI 投资建议、自动财务报告生成
8. 短信、邮件、推送等复杂通知系统

### 2.4 一句话目标
Smart Finance Dashboard V1 的目标是：

**让用户通过导入账单文件，快速获得自动分类、消费洞察和预算预警。**

## 三、核心用户场景与用户故事

### 3.1 核心用户场景

1. 用户希望快速整理最近一段时间的消费记录
用户已有银行或支付平台导出的账单文件，希望系统帮助其完成结构化整理。

2. 用户希望减少手工分类工作
用户不希望逐笔判断交易属于哪个消费类别，希望系统自动完成大部分分类。

3. 用户希望看懂自己的消费结构
用户想快速知道自己总共花了多少钱、主要花在哪些地方、哪些类别支出最高。

4. 用户希望控制消费节奏
用户希望设置总预算和分类预算，用于约束每月支出。

5. 用户希望在超支前获得提醒
用户希望系统在预算接近上限或已经超支时给出明确提示。

### 3.2 核心用户任务

1. 上传账单文件并完成导入
2. 查看导入后的流水记录
3. 获取自动分类结果
4. 修正错误分类或补充规则
5. 查看消费分析看板
6. 设置预算并查看预算状态
7. 识别超支或接近超支项目

### 3.3 用户故事

1. 作为一个不愿意手工记账的普通用户，我希望上传已有账单文件后，系统能自动解析交易数据，这样我就不用逐条录入消费记录。
2. 作为一个想快速了解花销去向的用户，我希望系统自动把交易归入餐饮、交通、购物等类别，这样我能直接看到消费结构。
3. 作为一个对自动分类结果有要求的用户，我希望在识别错误时手动修正分类，并补充自己的关键词规则，这样以后类似交易可以被更准确地识别。
4. 作为一个关注月度花销的用户，我希望看到本月总支出、总收入、净结余和分类占比，这样我能快速判断支出是否合理。
5. 作为一个有预算意识的用户，我希望设置月总预算和分类预算，这样我能更好地控制消费。
6. 作为一个希望及时调整支出的用户，我希望在预算接近上限或超支时收到明显提醒，这样我能在月底之前采取行动。

### 3.4 典型使用流程

1. 用户首次进入系统
2. 用户上传账单文件
3. 系统解析并标准化交易数据
4. 系统基于默认规则完成初步分类
5. 用户查看导入结果和流水明细
6. 用户修正少量误分类交易或补充规则
7. 系统生成月度消费看板
8. 用户设置月总预算和分类预算
9. 系统展示预算执行状态和预警信息

### 3.5 低频但必须考虑的场景

1. 导入文件格式不符合要求
2. 导入内容缺少关键字段
3. 导入疑似重复账单
4. 存在大量未分类交易
5. 用户尚未设置预算但希望先看消费分析

## 四、功能需求

### 4.1 功能模块总览
V1 拆分为以下 5 个核心模块：

1. 账单导入模块
2. 交易流水与分类模块
3. 消费分析看板模块
4. 预算与预警模块
5. 基础系统与用户体验模块

### 4.2 账单导入模块

#### 功能说明
该模块负责把外部账单文件转为系统可识别、可分析的交易数据，是整个产品的入口。

#### 功能要求

1. 支持用户上传账单文件
2. 支持解析账单中的关键字段，包括交易时间、金额、商户、收支方向、备注等
3. 支持导入结果反馈，包括成功条数、失败条数、异常提示、疑似重复提示
4. 支持导入失败时展示明确原因
5. 导入成功后支持进入流水页或看板页继续操作

#### 优先级
- P0：文件上传、基础解析、导入结果反馈
- P1：疑似重复提示、导入说明优化
- P2：更多平台模板适配

#### V1 限制

1. 不要求对接第三方平台 API
2. 不要求支持图片、PDF、截图识别
3. 不要求复杂平台模板全覆盖

### 4.3 交易流水与分类模块

#### 功能说明
该模块负责展示导入后的交易数据，并完成自动分类与人工修正。

#### 功能要求

1. 支持交易流水列表展示
2. 支持自动分类
3. 支持默认规则命中逻辑
4. 支持“未分类”状态
5. 支持人工修正分类
6. 支持用户补充规则
7. 支持规则的新增、查看、删除

#### 优先级
- P0：流水展示、自动分类、未分类标记
- P1：人工修正分类、用户补充规则
- P2：复杂规则能力

#### V1 限制

1. 不做复杂优先级体系
2. 不做高级规则编辑器
3. 不做自动学习推荐规则

### 4.4 消费分析看板模块

#### 功能说明
该模块负责将流水数据和分类结果转化为用户可理解的可视化分析。

#### 功能要求

1. 展示月度总览，包括总支出、总收入、净结余、交易笔数
2. 展示分类消费分布
3. 展示时间趋势分析
4. 展示重点分类排名
5. 支持时间范围筛选
6. 支持未分类交易提示入口

#### 优先级
- P0：月度总览、分类分布、预算状态展示
- P1：趋势分析、时间筛选、未分类提醒
- P2：更多图表形式和深入分析

#### V1 限制

1. 不做复杂 BI 自定义分析
2. 不做 AI 财务报告生成
3. 不做长期预测分析

### 4.5 预算与预警模块

#### 功能说明
该模块负责帮助用户建立支出边界，并在预算接近或超出时给出提示。

#### 功能要求

1. 支持设置月总预算
2. 支持设置分类预算
3. 展示预算金额、已用金额、剩余金额、使用比例和状态
4. 支持预算预警
5. 在首页看板中展示预算状态

#### 优先级
- P0：月总预算、分类预算、预算状态展示
- P1：预算预警高亮与入口联动
- P2：更复杂预算策略

#### V1 限制

1. 不做按周预算
2. 不做自定义复杂预算周期
3. 不做自动预算推荐

### 4.6 基础系统与用户体验模块

#### 功能说明
该模块负责保障产品基础可用性和使用体验。

#### 功能要求

1. 支持响应式 Web 界面
2. 支持清晰的信息架构
3. 支持空状态展示
4. 支持错误状态提示
5. 保证不同页面的数据一致性

#### 优先级
- P0：核心页面、基础空状态、错误状态
- P1：更完整的新手引导与状态提示
- P2：更细致的体验优化

## 五、关键业务规则与数据规则

### 5.1 交易数据标准结构
每条交易记录至少应包含以下字段：

1. 交易时间
2. 交易金额
3. 收支方向
4. 交易对象或商户名称
5. 交易备注或摘要
6. 分类结果
7. 数据来源

### 5.2 导入处理规则

1. 不同来源的账单字段需要映射为统一内部字段
2. 缺失关键字段的记录不能进入正式统计
3. 同一交易不能重复计入统计
4. 所有导入数据需先标准化后再进入分类和统计流程

### 5.3 分类规则

1. 分类依据主要包括商户名称、备注、摘要、关键词规则
2. V1 提供固定基础分类：
   - 餐饮
   - 交通
   - 购物
   - 居家日用
   - 医疗健康
   - 娱乐休闲
   - 其他
   - 未分类
3. 默认规则优先覆盖常见生活消费
4. 用户规则用于补足个性化分类需求
5. 未命中任何规则的交易归入“未分类”

### 5.4 人工修正规则

1. 用户可对单笔交易修改分类
2. 人工修正结果优先于自动分类
3. 用户补充的规则用于后续交易分类
4. 修改后的结果应同步反映到看板和预算统计中

### 5.5 时间归属规则

1. 交易归属周期以交易日期为准
2. 月度统计按自然月计算
3. 预算归属按自然月计算

### 5.6 收支统计规则

1. 总支出只统计支出类交易
2. 总收入只统计收入类交易
3. 净结余 = 总收入 - 总支出
4. 分类分布默认以支出类交易为主

### 5.7 预算计算规则

1. 月总预算针对当月全部支出
2. 分类预算仅针对对应分类支出
3. 未设置预算的分类只参与统计，不参与预警
4. 总预算和分类预算可以同时存在

### 5.8 预算预警规则

1. 预算使用率 < 80%：正常
2. 预算使用率 >= 80% 且 < 100%：接近预算
3. 预算使用率 >= 100%：已超支

### 5.9 未分类处理规则

1. 未分类交易必须可见、可筛选、可修正
2. 首页或看板需要有未分类提醒入口
3. 未分类交易不能被默认隐藏

### 5.10 数据一致性原则

1. 同一笔交易在各页面展示口径必须一致
2. 分类修改后，看板与预算统计应同步更新
3. 统计结果必须可以回溯到明细记录

## 六、页面与交互需求

### 6.1 页面结构
V1 包含以下核心页面：

1. 首页看板
2. 账单导入页
3. 流水明细页
4. 规则管理页
5. 预算设置页

### 6.2 首页看板

#### 页面目标
让用户快速回答以下问题：

1. 本月花了多少钱
2. 主要花在哪些类别
3. 预算是否快超了
4. 是否存在未分类交易

#### 页面内容

1. 月度关键指标区
2. 分类消费分布区
3. 支出趋势区
4. 预算状态区
5. 未分类提示区
6. 时间筛选区

#### 交互要求

1. 优先展示结论信息
2. 支持从看板跳转到对应明细
3. 无数据时明确引导用户上传账单

### 6.3 账单导入页

#### 页面目标
帮助用户低门槛完成账单上传并获得明确反馈。

#### 页面内容

1. 上传区域
2. 导入说明区域
3. 导入结果反馈区域
4. 后续行动入口

#### 交互要求

1. 上传流程简单直接
2. 上传中有处理状态反馈
3. 导入失败时有明确错误提示
4. 未导入数据时承担新手引导作用

### 6.4 流水明细页

#### 页面目标
用于查看交易记录、处理未分类和修正分类。

#### 页面内容

1. 交易列表
2. 筛选与搜索区域
3. 分类编辑能力
4. 未分类处理入口

#### 交互要求

1. 分类修改应轻量高效
2. 筛选结果应即时生效
3. 修改后需要明确反馈
4. 修改结果需自动同步到看板和预算统计

### 6.5 规则管理页

#### 页面目标
帮助用户查看已有规则并补充自己的简单分类规则。

#### 页面内容

1. 规则列表
2. 新增规则入口
3. 基础管理能力

#### 交互要求

1. 规则新增流程尽量简单
2. 规则逻辑需要可理解
3. 无规则时需提供空状态说明

### 6.6 预算设置页

#### 页面目标
帮助用户设置预算并查看预算执行情况。

#### 页面内容

1. 月总预算设置区
2. 分类预算设置区
3. 预算状态展示区

#### 交互要求

1. 预算设置流程简单
2. 未设置某分类预算时不影响其他预算展示
3. 无预算时提供引导说明

### 6.7 空状态要求

系统至少需要覆盖以下空状态：

1. 无账单数据
2. 无预算配置
3. 无用户规则
4. 当前时间范围无数据

空状态页面需要同时包含说明与下一步动作入口。

### 6.8 错误状态要求

1. 导入失败需说明原因
2. 文件格式不支持需提示支持范围
3. 数据为空或解析失败需提示检查内容
4. 分类保存或预算保存失败时应支持重试

### 6.9 响应式要求

1. 桌面端为主要设计基准
2. 移动端保证可读、可滚动、可完成核心操作
3. 小屏设备上可适当简化复杂图表展示

### 6.10 页面交互原则

1. 主路径优先
2. 少配置先见价值
3. 结果可追溯
4. 状态明确

## 七、非功能需求与验收标准

### 7.1 非功能需求

1. 可用性
用户能完成从导入到账单分析的完整主流程。

2. 易用性
页面文案通俗、操作简单、无需复杂前置学习。

3. 性能表现
在个人常见账单规模下，导入、看板加载、流水浏览应保持流畅。

4. 数据准确性
不同页面展示口径一致，分类修正后统计结果同步更新。

5. 可维护性
模块边界清晰，规则体系和页面结构支持后续扩展。

6. 基础安全性
账单数据属于敏感信息，应避免无关暴露，遵循最小必要展示原则。

### 7.2 核心验收标准

1. 用户能够上传账单文件并获得明确导入结果
2. 导入后的交易可在流水页中查看
3. 系统能够对常见交易进行自动分类
4. 用户能够手动修正分类
5. 用户能够新增基础规则
6. 首页看板能够展示关键财务指标和分类分布
7. 用户能够设置月总预算和分类预算
8. 系统能够正确展示预算执行状态与超支提醒
9. 空状态和错误状态具备明确引导
10. 桌面端完整可用，移动端基本可用

### 7.3 模块验收口径

1. 账单导入模块
上传可识别文件后可生成可查看的交易记录；失败时能提示原因。

2. 流水与分类模块
交易记录可浏览、可筛选、可修改分类；自动分类结果可被修正。

3. 看板模块
系统能正确汇总并展示总览指标、分类分布和趋势信息。

4. 预算模块
用户可设置预算，系统可正确计算执行进度与预警状态。

5. 规则模块
用户可新增、查看、删除规则，且规则对后续分类有效。

### 7.4 暂不纳入 V1 验收范围

1. 高级规则引擎能力
2. 极高自动分类准确率承诺
3. 短信、邮件、推送通知
4. 借贷、理财、资产等复杂财务模块
5. 企业级并发和多租户能力

### 7.5 产品成功判断标准

1. 用户能够在较短时间内完成一次账单导入
2. 系统能够给出基础可用的自动分类结果
3. 用户能够直接看到消费结构与预算状态
4. 用户能够在发现错误后修正分类并补充规则
5. 系统能清晰提示预算接近上限或已经超支
6. 用户不需要专业财务知识也能完成主流程

## 八、版本总结与后续迭代方向

### 8.1 V1 总结
Smart Finance Dashboard 第一版的核心目标是：

**让用户通过导入账单文件，快速获得自动分类、消费洞察和预算预警。**

V1 重点不是做一个大而全的财务平台，而是先做出一个对普通用户真正有价值的轻量型产品。

### 8.2 V1 主链路

1. 上传账单文件
2. 系统解析并标准化交易数据
3. 自动完成初步分类
4. 用户查看流水和消费看板
5. 用户修正分类或补充规则
6. 用户设置预算
7. 系统展示预算状态和超支预警

### 8.3 后续迭代方向

1. 支持更多账单模板与平台适配
2. 增强规则引擎能力
3. 增加更丰富的数据分析能力
4. 提供更智能的分类优化体验
5. 完善预算体系
6. 引入账户、云端存储、数据备份等能力

### 8.4 产品结论
V1 的判断标准可以总结为：

**少手工记账、能看懂消费、能提醒超支。**

只要产品能够稳定支撑这条主链路，就具备成为可用个人财务产品的基础。
