## 文档摘要

本文档用于承接 Smart Finance Dashboard 的 V1 PRD，将需求进一步落为可实施的产品与技术总方案。文档重点解决以下问题：

1. V1 的系统应该如何分层和组织模块
2. 页面、模块、数据流、接口和核心实体如何对应
3. 前后端各自承担什么职责
4. V1 的关键实现边界、扩展点和工程约束是什么
5. 开发工作应按什么顺序推进，才能稳定打通主链路

本方案默认采用以下前提：

1. 前后端分离 Web 应用
2. 数据库持久化
3. 统一交易结构 + 平台适配层
4. 简单规则引擎并预留优先级扩展

## 一、总体方案设计

### 1.1 文档定位
本方案设计文档位于 PRD 和后续开发计划之间，作用不是重复需求文档，也不是直接展开到底层代码实现，而是回答一个更关键的问题：

`如何把已经明确的产品需求，落成一个可实施、可扩展、边界清晰的 V1 系统方案。`

因此，这份文档同时覆盖两层内容：

1. 产品结构层
定义页面、模块、用户主流程和功能之间的承接关系。

2. 系统实现层
定义前后端职责、核心服务模块、数据流、存储方式和关键实现策略。

### 1.2 方案目标
V1 总方案需要达成以下目标：

1. 支撑完整主链路
从“上传账单”到“自动分类”再到“看板分析”和“预算预警”，整条链路闭环。

2. 保证系统结构清晰
前端、后端、数据存储、规则处理、统计分析之间职责边界明确。

3. 支持 V1 快速落地
方案不为“未来很强大”引入过重设计，以第一版稳定交付为核心目标。

4. 为后续扩展预留空间
虽然 V1 功能受控，但架构上应能平滑扩展到账单模板增加、规则增强、图表扩展和预算体系升级。

### 1.3 核心设计原则
整个 V1 方案建议遵循以下 6 条原则：

1. 主链路优先
任何模块设计都优先服务“导入 -> 分类 -> 看板 -> 预算”主流程。

2. 单一职责
导入解析、分类规则、统计聚合、预算计算、前端展示应分成边界清楚的模块。

3. 结果可追溯
看板指标、预算状态、分类结果都必须能回溯到交易明细。

4. 简单可扩展
V1 只实现必要能力，但模块接口和数据结构为未来扩展留出位置。

5. 前后端职责清晰
前端负责交互承载与可视化展示，后端负责导入解析、规则匹配、数据持久化、统计汇总与预算计算。

6. 规则可解释、可修正
自动分类不追求“神奇智能”，而追求“规则透明、结果可信、允许用户纠正”。

### 1.4 总体架构视图
V1 建议采用标准的前后端分离 Web 架构，由 4 层组成：

1. 前端应用层
负责页面展示、用户交互、图表渲染、筛选操作、表单提交和状态反馈。

2. 后端业务层
负责账单导入、交易标准化、分类规则处理、预算计算、统计聚合和接口输出。

3. 数据持久层
负责保存交易记录、分类规则、预算配置和导入任务结果等核心业务数据。

4. 文件处理层
负责接收用户上传的账单文件，并交由对应平台适配器解析为统一交易结构。

整体逻辑如下：

```text
前端页面
→ 调用后端接口
→ 后端完成导入/分类/统计/预算处理
→ 数据库存储结构化结果
→ 前端读取聚合结果并展示
```

### 1.5 前后端职责划分
前端负责：

1. 账单上传入口与导入结果展示
2. 看板页面和图表展示
3. 流水列表展示、筛选和分类编辑交互
4. 规则管理页面和预算设置页面
5. 错误状态、空状态、加载状态的表达

后端负责：

1. 上传文件接收与解析调度
2. 不同平台账单模板适配
3. 统一交易结构生成
4. 自动分类规则执行
5. 交易数据持久化
6. 看板统计聚合
7. 预算进度和预警计算
8. 对前端输出统一接口

关键原则：

`前端不承担业务真相计算，后端才是分类、统计和预算口径的唯一来源。`

### 1.6 V1 核心系统闭环
从系统视角看，V1 的核心闭环不是页面，而是数据流闭环：

1. 用户上传账单文件
2. 后端识别文件类型并进入对应解析器
3. 解析器输出统一格式的交易数据
4. 分类引擎根据默认规则和用户规则进行分类
5. 后端将交易、分类结果和导入信息写入数据库
6. 统计服务基于交易数据生成看板指标
7. 预算服务基于交易分类结果计算预算执行状态
8. 前端读取看板、流水、预算数据并展示给用户
9. 用户修正分类或新增规则后，后端重新更新相关统计结果

### 1.7 V1 非目标在架构层的体现
PRD 中的非目标，在方案层需要体现成明确的“不设计”约束：

1. 不设计多人体系
2. 不设计平台实时同步机制
3. 不设计复杂规则平台
4. 不设计理财/借贷域模型

这样做的意义是：

`防止方案阶段就把系统复杂度提前抬高。`

### 1.8 本部分结论
Smart Finance Dashboard V1 应被设计为一个以前后端分离为基础、以交易数据为核心、以分类结果为桥梁、以看板和预算为输出的轻量型财务分析系统。

## 二、产品结构与模块设计

### 2.1 产品结构总览
基于 PRD，V1 产品建议由 5 个用户可见模块和 4 个系统支撑模块共同组成。

用户可见模块：

1. 首页看板模块
2. 账单导入模块
3. 流水明细模块
4. 规则管理模块
5. 预算设置模块

系统支撑模块：

1. 文件解析与导入处理模块
2. 交易标准化与存储模块
3. 分类规则引擎模块
4. 统计与预算计算模块

### 2.2 产品模块与系统模块的映射关系
页面与业务逻辑不应绑死在一起。更合理的设计是：

1. 账单导入页对应后端的“文件解析与导入处理模块”
2. 流水明细页对应后端的“交易标准化与存储模块”以及“分类规则引擎模块”
3. 首页看板对应后端的“统计与预算计算模块”
4. 规则管理页对应后端的“分类规则引擎模块”
5. 预算设置页对应后端的“统计与预算计算模块”以及预算配置存储能力

### 2.3 首页看板模块设计
模块职责：

1. 展示月度关键指标
2. 展示分类消费结构
3. 展示消费趋势
4. 展示预算状态与预警
5. 展示未分类提醒和待处理入口

设计重点：

1. 首页只负责“看结果”和“跳转处理”
2. 首页不承担复杂编辑逻辑
3. 首页展示的数据必须来自统一统计口径
4. 首页中的每项结论应支持回溯到明细页

### 2.4 账单导入模块设计
模块职责：

1. 接收用户上传文件
2. 将文件发送给后端进行识别和解析
3. 展示导入处理中的状态
4. 展示导入结果摘要
5. 将用户引导到流水页或看板页

设计重点：

1. 导入流程必须短，不能让用户先做大量配置
2. 导入结果必须明确，包括成功条数、失败条数、异常原因
3. 导入只是数据进入系统的入口，不承担分类和统计展示职责

### 2.5 流水明细模块设计
模块职责：

1. 展示标准化后的交易明细
2. 支持按时间、分类、关键词等条件筛选
3. 支持查看未分类交易
4. 支持手动修正交易分类
5. 提供进入规则管理的延伸入口

设计重点：

1. 明细页必须可追踪每一笔交易
2. 自动分类结果必须可修正
3. 修正动作必须轻量
4. 明细页是看板与规则管理之间的桥梁

### 2.6 规则管理模块设计
模块职责：

1. 展示用户自定义规则
2. 支持新增规则
3. 支持删除规则
4. 让用户理解规则会如何影响后续分类结果

设计重点：

1. V1 只做简单规则管理
2. 规则尽量面向“关键词/商户 -> 分类”的清晰映射
3. 规则管理页应低频但高价值

### 2.7 预算设置模块设计
模块职责：

1. 设置月总预算
2. 设置分类预算
3. 查看预算执行情况
4. 查看当前预警状态

设计重点：

1. 预算配置必须简单直接
2. 总预算和分类预算并行存在
3. 预算页既是配置页，也是预算状态查看页
4. 预算结果要与首页保持一致

### 2.8 文件解析与导入处理模块设计
模块职责：

1. 接收上传文件
2. 识别文件来源或模板结构
3. 调用对应平台解析器
4. 输出统一交易结构
5. 生成导入结果摘要
6. 将数据交给后续分类和存储流程

设计重点：

1. 采用“统一交易结构 + 平台适配层”模式
2. 平台差异应被隔离在解析器内部
3. 上层业务逻辑不应依赖具体平台文件字段

### 2.9 交易标准化与存储模块设计
模块职责：

1. 接收解析后的交易数据
2. 标准化字段口径
3. 处理基础数据清洗
4. 保存交易记录与导入来源信息
5. 为分类、统计和预算提供统一数据基础

设计重点：

1. 所有后续能力都依赖这个统一事实层
2. 原始平台差异必须在这里被消化
3. 交易数据一旦入库，应成为后续统计的唯一基础来源

### 2.10 分类规则引擎模块设计
模块职责：

1. 维护默认规则与用户规则
2. 按既定匹配顺序执行分类
3. 输出自动分类结果
4. 支持人工修正后的结果覆盖自动结果
5. 为后续统计和预算提供最终分类结果

设计重点：

1. V1 采用简单规则引擎
2. 当前实现以关键词/商户匹配为主
3. 设计上预留未来优先级扩展能力
4. 人工修正结果始终优先于自动规则结果

### 2.11 统计与预算计算模块设计
模块职责：

1. 统计总收入、总支出、净结余、交易笔数
2. 聚合分类分布和趋势数据
3. 计算月总预算执行情况
4. 计算分类预算执行情况
5. 输出预算状态和预警标识

设计重点：

1. 所有统计口径必须统一
2. 看板和预算共享同一套分类与交易事实层
3. 用户修改分类后，该模块应能基于最新数据重新输出结果

### 2.12 模块协作关系
V1 模块协作建议按下面这条主链路理解：

```text
账单导入模块
→ 文件解析与导入处理模块
→ 交易标准化与存储模块
→ 分类规则引擎模块
→ 统计与预算计算模块
→ 首页看板 / 流水明细 / 预算设置
```

纠偏路径：

```text
流水明细模块
→ 人工修正分类
→ 更新最终分类结果
→ 触发统计与预算结果更新
→ 首页看板与预算页读取最新结果
```

规则增强路径：

```text
规则管理模块
→ 新增用户规则
→ 后续新导入交易使用新规则分类
```

### 2.13 本部分结论
V1 的产品结构应以前台页面承载用户操作、以后端模块承担业务真相、以交易数据和分类结果串联整个系统。

## 三、核心业务流程与数据流设计

### 3.1 设计目标
这一部分的目标是回答两个问题：

1. 用户完成一次核心操作时，系统内部到底发生了什么
2. 哪些数据是原始事实，哪些数据是处理结果，哪些数据是聚合输出

V1 中，系统应围绕“交易数据”建立统一数据流。所有看板、预算、分类和预警，最终都要以标准化交易数据为基础。

### 3.2 全局数据流总览
V1 的核心数据流建议设计为：

```text
账单文件
→ 文件解析
→ 交易标准化
→ 自动分类
→ 交易入库
→ 统计聚合
→ 预算计算
→ 前端展示
→ 用户修正分类 / 新增规则
→ 更新交易结果与统计结果
```

这条链路里有 3 个关键层次：

1. 原始输入层
账单文件及其解析结果，是系统的数据入口。

2. 事实数据层
标准化交易记录和最终分类结果，是系统的业务真相层。

3. 聚合输出层
看板指标、趋势数据、预算状态、预警标识，是面向页面展示的计算结果层。

### 3.3 账单导入主流程
流程定义：

1. 用户在前端上传账单文件
2. 前端将文件发送到后端导入接口
3. 后端识别文件类型或模板结构
4. 后端选择对应平台解析器进行解析
5. 解析器提取基础字段并输出标准化候选记录
6. 系统对候选记录进行字段校验与标准化处理
7. 系统识别异常记录和疑似重复记录
8. 合法交易记录进入自动分类流程
9. 分类完成后写入数据库
10. 后端返回导入结果摘要给前端
11. 前端展示导入结果，并引导用户进入流水页或看板页

设计重点：

1. 导入流程必须是异步感知式的
2. 导入结果必须带结构化反馈
3. 导入完成后应立即进入价值展示阶段

### 3.4 文件解析与标准化流程
建议流程：

1. 文件接收层拿到上传文件
2. 模板识别器判断该文件更接近哪种账单格式
3. 对应解析器读取文件内容
4. 解析器将平台字段映射成统一候选字段
5. 标准化模块统一处理：
   - 时间格式
   - 金额格式
   - 收支方向
   - 商户名称
   - 备注摘要
6. 标准化模块输出统一交易结构

关键原则：

1. 解析器负责“适配平台差异”
2. 标准化模块负责“统一系统口径”
3. 分类、统计、预算都只能消费标准化后的交易结构

### 3.5 自动分类流程
建议流程：

1. 标准化交易记录进入分类引擎
2. 分类引擎按既定顺序加载规则
3. 先匹配内置默认规则
4. 再匹配用户自定义规则，或按约定优先顺序合并判断
5. 若命中规则，则生成自动分类结果
6. 若未命中任何规则，则标记为“未分类”
7. 分类结果与交易记录一并写入数据库

V1 设计建议：

1. 采用简单可解释的匹配逻辑
2. 输出“自动分类结果”与“最终分类结果”两个概念
3. 未分类是正常状态

### 3.6 人工修正分类流程
建议流程：

1. 用户在流水页查看交易记录
2. 用户发现分类错误或未分类交易
3. 用户修改该笔交易的分类
4. 前端向后端提交更新请求
5. 后端将该笔交易的最终分类更新为人工修正结果
6. 后端重新触发相关统计与预算计算
7. 前端刷新该交易及相关看板/预算数据

设计重点：

1. 人工修正结果必须覆盖自动分类结果
2. 修正动作应尽量作用于单条交易
3. 修正后带来的统计变化必须即时反映

### 3.7 用户新增规则流程
建议流程：

1. 用户进入规则管理页
2. 输入关键词或商户匹配文本
3. 选择目标分类
4. 前端提交规则创建请求
5. 后端校验规则合法性并保存
6. 新规则进入后续分类规则集合
7. 之后新导入的交易使用新规则进行匹配

V1 设计建议：

1. 新增规则默认优先作用于“后续新导入交易”
2. 是否回溯重算历史交易，不作为 V1 必选能力
3. 规则新增流程必须简单

### 3.8 首页看板生成流程
建议流程：

1. 前端请求某一时间范围的看板数据
2. 后端读取该时间范围内的标准交易数据和最终分类结果
3. 统计模块计算：
   - 总收入
   - 总支出
   - 净结余
   - 交易笔数
4. 分类聚合模块计算：
   - 各分类金额
   - 各分类占比
   - 分类排名
5. 趋势模块计算按时间粒度聚合的时间序列
6. 预算模块计算当前预算状态
7. 后端将结果组装成看板响应对象
8. 前端按模块展示

### 3.9 预算计算与预警流程
建议流程：

1. 用户设置月总预算与分类预算
2. 预算数据写入数据库
3. 当前周期内的支出交易进入预算计算逻辑
4. 系统分别计算：
   - 月总预算已使用金额
   - 各分类预算已使用金额
5. 系统计算使用比例
6. 根据阈值判定状态：
   - 正常
   - 接近预算
   - 已超支
7. 结果同时提供给首页看板与预算页

关键原则：

1. 预算计算只基于支出类交易
2. 分类预算只基于最终分类结果
3. 用户修改分类后，预算状态应随之更新
4. 没设置预算的分类，不参与预警，只参与看板统计

### 3.10 查询与筛选数据流
流水页查询数据流：

1. 前端提交筛选条件
2. 后端根据条件查询交易记录
3. 后端返回分页或列表结果
4. 前端渲染列表

看板筛选数据流：

1. 前端切换时间范围
2. 后端重新计算或查询对应统计结果
3. 前端更新图表和指标卡片

设计重点：

1. 筛选条件应始终由后端解释
2. 前端不保存“计算后的真相”
3. 同样的筛选口径在不同页面应尽量复用

### 3.11 错误流与异常处理流
导入异常流：

1. 文件格式无法识别
2. 关键字段缺失
3. 文件内容为空
4. 解析失败
5. 疑似重复数据过多

分类异常流：

1. 某些交易无法分类
2. 规则不命中
3. 用户规则配置冲突或无效

预算异常流：

1. 未设置预算
2. 某分类没有预算配置
3. 当前月份无支出数据

这些情况不应中断主流程，而应转化为用户可理解的业务状态。

### 3.12 数据一致性流
V1 必须确保以下一致性链路：

1. 导入完成后，流水页看到的记录应是统计和预算的共同来源
2. 流水页中某笔交易分类修改后，看板和预算必须基于新结果刷新
3. 规则新增后，新导入数据的分类行为应与规则管理页中看到的规则保持一致
4. 首页和预算页展示的预算状态必须来自同一套计算逻辑

### 3.13 本部分结论
V1 的系统核心不是页面，而是围绕标准化交易记录建立一条从导入、分类、修正到聚合输出的统一数据流。

## 四、系统架构与技术分层设计

### 4.1 设计目标
需要解决的问题：

1. 系统应该如何分层，避免后续代码结构混乱
2. 前端、后端、数据库和文件处理之间如何协作
3. 哪些能力应该做成独立模块
4. V1 如何在不过度设计的前提下具备后续可扩展性

V1 采用清晰、稳定、可快速交付的分层式单后端架构。

### 4.2 总体技术架构
总体架构建议如下：

```text
前端 Web 应用
    ↓ HTTP / JSON API
后端应用服务
    ├─ 导入与文件解析模块
    ├─ 交易管理模块
    ├─ 分类规则模块
    ├─ 看板统计模块
    ├─ 预算计算模块
    └─ 查询与管理接口层
    ↓
关系型数据库
```

逻辑上可拆为 5 层：

1. 表现层
2. 接口层
3. 业务服务层
4. 数据访问层
5. 基础设施层

### 4.3 前端架构设计
前端职责：

1. 页面路由与导航
2. 文件上传与表单交互
3. 看板图表展示
4. 流水列表筛选与编辑交互
5. 预算和规则配置页面
6. 与后端 API 的通信
7. 加载、空状态、错误状态展示

前端模块建议：

1. 页面层
2. 组件层
3. 服务层
4. 状态层

前端设计原则：

1. 前端不持有业务真相
2. 前端尽量不重复实现业务计算
3. 前端主要负责交互和展示

### 4.4 后端架构设计
后端建议模块：

1. 导入模块
2. 交易模块
3. 分类模块
4. 看板模块
5. 预算模块
6. 规则模块

后端设计原则：

1. 模块间通过明确服务接口协作
2. 统计和预算依赖交易与最终分类结果
3. 导入模块不直接承担页面逻辑
4. 控制器层只做请求编排

### 4.5 分层职责边界
Controller / API 层：

1. 接收请求
2. 做基础参数校验
3. 调用业务服务
4. 返回统一响应格式

Service 层：

1. 编排业务流程
2. 管理事务边界
3. 处理核心业务判断
4. 组织跨模块调用

Repository / DAO 层：

1. 负责数据库读写
2. 提供按条件查询能力
3. 封装持久化细节

基础设施层：

1. 文件读写
2. 日志记录
3. 错误码与异常处理
4. 配置管理
5. 时间与格式工具

### 4.6 文件处理架构设计
建议把文件处理拆成以下子层：

1. 上传接收层
2. 模板识别层
3. 平台解析器层
4. 标准化转换层

设计重点：

1. 模板识别和解析器要隔离
2. 业务模块只能拿到标准化后的结果
3. 新增平台模板时，应只新增适配器

### 4.7 分类引擎架构设计
分类引擎建议在后端内部作为独立业务模块存在。

建议拆分为：

1. 规则加载器
2. 规则匹配器
3. 分类结果生成器
4. 分类覆盖器

V1 架构重点：

1. 简单规则逻辑即可
2. 预留优先级字段或排序机制
3. 默认规则与用户规则应能区分来源
4. 最终分类结果必须有统一写入位置

### 4.8 看板统计架构设计
建议拆分为：

1. 总览指标服务
2. 分类聚合服务
3. 趋势统计服务
4. 看板响应组装服务

设计重点：

1. 所有统计服务共用同一套筛选条件定义
2. 所有统计服务都基于统一交易事实层
3. 看板接口应偏“前端可消费”

### 4.9 预算服务架构设计
建议拆分为：

1. 预算配置服务
2. 预算计算服务
3. 预算状态判定服务
4. 预算响应组装服务

设计重点：

1. 预算配置和预算结果要区分开
2. 预算状态由服务端统一计算
3. 首页和预算页共享同一套预算计算结果来源

### 4.10 数据库架构设计
数据库主要承载 4 类核心数据：

1. 交易数据
2. 导入记录数据
3. 分类规则数据
4. 预算配置数据

数据库设计原则：

1. 交易是核心事实表
2. 分类结果应能表达自动与人工修正状态
3. 导入记录应能帮助追踪来源和问题
4. 预算和规则应与交易领域解耦但可关联

### 4.11 API 设计风格
设计原则：

1. 使用 REST 风格即可
2. 页面以资源查询和资源更新为主
3. 文件上传使用单独上传接口
4. 统计与看板使用聚合查询接口
5. 统一响应结构

API 大类建议：

1. 导入接口
2. 交易查询与分类修改接口
3. 规则管理接口
4. 预算配置接口
5. 看板统计接口

### 4.12 错误处理架构
建议统一三类错误：

1. 用户可理解错误
2. 业务状态型结果
3. 系统异常

设计上需要保证：

1. 用户可理解错误明确返回给前端
2. 业务状态不要伪装成系统错误
3. 系统异常统一记录日志并返回通用错误信息

### 4.13 可扩展性设计
需要预留扩展点：

1. 文件解析器可扩展
2. 分类规则可扩展
3. 看板统计可扩展
4. 预算体系可扩展

这里的“预留扩展”不是现在就实现，而是现在不要把结构写死。

### 4.14 本部分结论
V1 应采用前后端分离 + 单体后端分层架构，以清晰模块边界替代复杂分布式设计，以统一交易事实层支撑分类、看板和预算三个核心能力。

## 五、核心数据模型设计

### 5.1 设计目标
V1 的数据模型设计需要同时满足 4 个目标：

1. 支撑主链路闭环
2. 保证数据口径统一
3. 支持人工纠偏
4. 为未来扩展留有空间

### 5.2 核心实体总览
V1 建议围绕以下 6 个核心实体建模：

1. `ImportJob`
2. `ImportedFile`
3. `Transaction`
4. `CategoryRule`
5. `BudgetPlan`
6. `BudgetSnapshot` 或预算计算结果对象

真正的核心实体是 `Transaction`。

### 5.3 ImportJob 设计
建议承载的信息包括：

1. 导入任务标识
2. 导入开始时间 / 完成时间
3. 导入状态
4. 成功记录数
5. 失败记录数
6. 疑似重复记录数
7. 错误摘要信息
8. 数据来源类型

设计意义：

1. 让系统可以追踪某次导入发生了什么
2. 让前端可以展示结构化导入结果
3. 为后续排查问题提供任务级上下文

### 5.4 ImportedFile 设计
建议承载的信息包括：

1. 文件标识
2. 原始文件名
3. 文件类型
4. 文件大小
5. 上传时间
6. 对应导入任务 ID
7. 文件来源平台或模板类型
8. 文件存储位置或引用标识

V1 如果想控制实现复杂度，也可以让 `ImportedFile` 内聚在 `ImportJob` 中，但方案层建议保留它作为独立概念。

### 5.5 Transaction 设计
建议包含的核心信息：

1. 交易标识
2. 所属导入任务 ID
3. 交易日期
4. 交易金额
5. 收支方向
6. 商户名称或交易对象
7. 交易摘要 / 备注
8. 原始来源文本信息
9. 自动分类结果
10. 最终分类结果
11. 分类来源状态
12. 是否为疑似重复
13. 创建时间 / 更新时间

设计重点：

1. `交易日期` 是周期归属基础
2. `金额 + 收支方向` 是统计与预算的基础
3. `商户名称 / 摘要` 是规则引擎的核心输入
4. `自动分类结果` 和 `最终分类结果` 应区分
5. `分类来源状态` 应能表达自动分类、人工修正、未分类

### 5.6 CategoryRule 设计
建议承载的信息包括：

1. 规则标识
2. 规则名称或描述
3. 匹配文本
4. 匹配类型
5. 目标分类
6. 规则来源
7. 启用状态
8. 排序值或优先顺序字段
9. 创建时间 / 更新时间

V1 设计重点：

1. 规则来源至少区分系统默认规则和用户自定义规则
2. 匹配类型先保持简单
3. 排序值字段建议预留

### 5.7 BudgetPlan 设计
建议承载的信息包括：

1. 预算标识
2. 预算周期
3. 预算类型
4. 目标分类
5. 预算金额
6. 状态
7. 创建时间 / 更新时间

预算类型建议至少支持：

1. 月总预算
2. 分类预算

### 5.8 BudgetSnapshot / 预算结果对象设计
建议承载的信息包括：

1. 预算配置标识
2. 当前已使用金额
3. 剩余金额
4. 使用比例
5. 当前状态
6. 对应时间范围

状态值建议：

1. 正常
2. 接近预算
3. 已超支

V1 更适合把它作为运行时聚合结果对象，而不是持久化表。

### 5.9 分类体系设计
V1 建议采用固定分类枚举或受控分类集合，例如：

1. 餐饮
2. 交通
3. 购物
4. 居家日用
5. 医疗健康
6. 娱乐休闲
7. 其他
8. 未分类

设计重点：

1. V1 分类体系固定
2. 规则和预算都依赖同一套分类定义
3. “未分类”必须是正式状态

### 5.10 关键实体关系
核心关系可以理解为：

```text
ImportJob
└─ ImportedFile

ImportJob
└─ Transaction (1:N)

CategoryRule
→ Transaction (通过分类执行产生结果，不直接外键绑定)

BudgetPlan
→ Transaction (通过分类和时间范围参与预算计算，不直接一对一绑定)
```

### 5.11 数据模型中的状态设计
建议重点明确以下状态：

`ImportJob` 状态：

1. 待处理
2. 处理中
3. 部分成功
4. 成功
5. 失败

`Transaction` 分类状态：

1. 自动分类
2. 人工修正
3. 未分类

`CategoryRule` 状态：

1. 启用
2. 停用

`Budget` 状态：

1. 正常
2. 接近预算
3. 已超支

### 5.12 字段设计原则

1. 金额字段必须使用适合财务计算的精确数值类型
2. 时间字段必须区分业务时间和系统时间
3. 分类结果不应只用一个模糊字段承载全部语义
4. 原始来源文本建议适度保留

### 5.13 数据模型扩展点

1. `Transaction` 未来可扩展更多来源字段
2. `CategoryRule` 未来可扩展更复杂条件
3. `BudgetPlan` 未来可扩展更多周期类型
4. 分类体系未来可扩展为配置化字典

### 5.14 本部分结论
V1 的数据模型应以 Transaction 为统一事实核心，以 ImportJob 记录导入过程，以 CategoryRule 管理分类逻辑，以 BudgetPlan 管理预算配置，并通过运行时聚合结果支撑看板与预算展示。

## 六、接口与前后端交互设计

### 6.1 设计目标
需要解决 4 个核心问题：

1. 前端每个页面需要调用哪些接口
2. 后端接口应该按什么资源边界组织
3. 哪些接口返回原始数据，哪些接口返回聚合结果
4. 如何保证前后端交互口径统一、易扩展、易维护

基本原则：

`页面围绕业务资源和聚合结果获取数据，而不是围绕数据库表直接暴露接口。`

### 6.2 接口分组建议
V1 建议将接口分成 5 组：

1. 导入接口
2. 交易接口
3. 规则接口
4. 预算接口
5. 看板接口

### 6.3 导入接口设计
建议接口能力：

1. 上传账单文件
2. 查询导入任务结果
3. 获取某次导入的摘要信息

建议职责划分：

上传接口用途：

1. 接收账单文件
2. 启动导入任务
3. 返回任务结果或任务标识

导入结果查询接口用途：

1. 查询某次导入任务的状态
2. 返回成功数、失败数、异常数、疑似重复数
3. 返回错误摘要信息

设计建议：

1. 若导入处理足够快，V1 可以直接同步返回导入结果
2. 若后续考虑更复杂解析流程，则建议保留任务查询接口能力
3. 前端不应直接解析文件内容

### 6.4 交易接口设计
建议接口能力：

1. 查询交易列表
2. 获取交易详情
3. 修改交易最终分类
4. 按筛选条件查询未分类交易

交易列表接口应支持的查询条件：

1. 时间范围
2. 分类
3. 分类状态
4. 关键词
5. 收支方向
6. 导入任务来源

返回数据建议包含：

1. 交易 ID
2. 交易日期
3. 金额
4. 收支方向
5. 商户名
6. 摘要
7. 自动分类
8. 最终分类
9. 分类来源状态
10. 是否疑似重复

分类修改接口用途：

1. 用户修改单笔交易最终分类
2. 后端更新最终分类结果
3. 触发后续看板与预算逻辑读取新结果

### 6.5 规则接口设计
建议接口能力：

1. 查询规则列表
2. 创建规则
3. 删除规则
4. 启用/停用规则

规则列表接口返回建议：

1. 规则 ID
2. 规则名称或描述
3. 匹配文本
4. 匹配类型
5. 目标分类
6. 规则来源
7. 状态
8. 排序值或优先级字段

规则创建接口输入建议：

1. 匹配文本
2. 匹配类型
3. 目标分类

### 6.6 预算接口设计
建议接口能力：

1. 查询当前周期预算配置
2. 创建或更新预算配置
3. 查询预算执行情况

预算结果接口应返回：

1. 预算类型
2. 对应分类
3. 预算金额
4. 已使用金额
5. 剩余金额
6. 使用比例
7. 当前状态

设计重点：

1. 预算配置和预算结果应逻辑分离
2. 前端预算页既要能读到配置，也要能读到结果
3. 首页预算卡片可复用预算结果接口，或由看板接口统一返回预算摘要

### 6.7 看板接口设计
更建议采用聚合看板接口方式，而不是前端分别请求很多碎片接口。

推荐模式：

一个首页看板接口，按时间范围返回：

1. 总览指标
2. 分类分布
3. 趋势数据
4. 预算摘要
5. 未分类摘要

这样做的好处：

1. 首页初始化时请求更简单
2. 后端可统一保证统计口径一致
3. 前端无需自己拼多个接口结果

### 6.8 统一响应结构设计
建议统一包含：

1. 是否成功
2. 业务码或错误码
3. 消息说明
4. 数据载荷

成功示例：

```json
{
  "success": true,
  "code": "OK",
  "message": "success",
  "data": {}
}
```

失败示例：

```json
{
  "success": false,
  "code": "IMPORT_FORMAT_INVALID",
  "message": "账单文件格式无法识别",
  "data": null
}
```

### 6.9 分页、筛选与排序策略
交易列表建议支持：

1. 分页参数
2. 时间范围筛选
3. 分类筛选
4. 状态筛选
5. 关键词筛选
6. 按时间倒序排序

设计建议：

1. 看板接口不需要分页
2. 交易列表接口建议分页
3. 排序能力保持简单，V1 先以时间排序为主

### 6.10 页面与接口映射关系
首页看板页依赖：

1. 看板聚合接口

账单导入页依赖：

1. 文件上传接口
2. 导入结果查询接口

流水明细页依赖：

1. 交易列表查询接口
2. 交易分类修改接口

规则管理页依赖：

1. 规则列表接口
2. 规则创建接口
3. 规则删除接口

预算设置页依赖：

1. 预算配置读取接口
2. 预算配置保存接口
3. 预算执行结果接口

### 6.11 接口幂等与一致性考虑
导入接口：

1. 用户重复上传相同文件时，应至少有疑似重复提示能力
2. 后端不能因为重复请求直接产生不可控重复交易

分类修改接口：

1. 同一笔交易重复提交相同分类，应视为幂等结果
2. 修改后系统读取到的结果必须立即一致

预算保存接口：

1. 同一周期预算重复保存时，应是更新语义
2. 月总预算和分类预算要有明确唯一性约束思路

### 6.12 前后端职责边界再确认
前端负责：

1. 组织页面交互
2. 管理当前页面状态
3. 调用接口并展示结果
4. 展示错误、加载和空状态

后端负责：

1. 接收输入
2. 校验业务参数
3. 执行文件解析、分类、统计、预算计算
4. 返回统一结构结果

关键原则：

1. 前端不重新计算预算比例
2. 前端不自行推导预算状态
3. 前端不自行做交易分类逻辑
4. 前端只渲染后端输出的业务结果

### 6.13 本部分结论
V1 的接口设计应围绕导入、交易、规则、预算和看板五类业务资源展开，其中看板接口采用聚合输出模式，交易接口采用查询与更新并行模式，前端只消费业务结果，不承担核心业务计算。

## 七、页面方案与前端交互设计

### 7.1 设计目标
这一部分需要解决的核心问题是：

1. 每个页面该展示什么
2. 每个页面不该承担什么
3. 页面内部模块如何布局与协作
4. 用户如何在页面之间完成主路径切换
5. 前端应如何组织页面交互与状态反馈

V1 的页面方案必须围绕一个原则：

`先让用户快速看到结果，再逐步进入修正和配置。`

### 7.2 页面结构总览
V1 维持 5 个核心页面：

1. 首页看板页
2. 账单导入页
3. 流水明细页
4. 规则管理页
5. 预算设置页

导航建议顺序：

1. 看板
2. 导入
3. 流水
4. 规则
5. 预算

### 7.3 首页看板页设计
页面目标：

1. 本月花了多少钱
2. 主要花在哪些类别
3. 预算是否正常
4. 是否还有待处理问题

页面建议布局：

1. 顶部筛选区
2. 核心指标卡片区
3. 分类消费分布区
4. 支出趋势区
5. 预算状态区
6. 待处理提醒区

首页不应承担的职责：

1. 不做规则编辑
2. 不做预算复杂配置
3. 不做大批量明细处理
4. 不做文件上传解析逻辑

### 7.4 账单导入页设计
页面目标：

帮助用户快速完成账单上传，并清楚知道结果如何。

页面建议布局：

1. 导入说明区
2. 文件上传区
3. 导入处理中状态区
4. 导入结果反馈区
5. 导入后跳转区

导入页不应承担的职责：

1. 不直接展示完整明细列表
2. 不直接做规则配置
3. 不直接承担统计看板功能

### 7.5 流水明细页设计
页面目标：

帮助用户查看交易真相、发现分类问题并快速纠正。

页面建议布局：

1. 顶部筛选与搜索区
2. 交易列表区
3. 单条交易分类编辑能力
4. 未分类快捷处理入口
5. 规则管理跳转入口

流水页不应承担的职责：

1. 不负责复杂规则批量编辑
2. 不负责预算配置
3. 不直接承担首页聚合统计展示

### 7.6 规则管理页设计
页面目标：

帮助用户以简单方式新增和管理规则，让系统后续分类更准确。

页面建议布局：

1. 规则说明区
2. 规则列表区
3. 新增规则区

规则页不应承担的职责：

1. 不做交易列表查看
2. 不做历史交易全量重算控制台
3. 不做复杂规则引擎编辑器

### 7.7 预算设置页设计
页面目标：

帮助用户建立预算边界，并清楚看到预算当前执行情况。

页面建议布局：

1. 月总预算配置区
2. 分类预算配置区
3. 当前预算状态区

预算页不应承担的职责：

1. 不做交易修正
2. 不做导入流程
3. 不做分类规则管理

### 7.8 页面之间的主路径跳转设计
首次使用主路径：

```text
首页空状态 / 导入引导
→ 导入页上传账单
→ 查看导入结果
→ 进入流水页确认数据
→ 返回首页看板看结果
→ 进入预算页设置预算
→ 返回首页观察预算状态
```

日常使用主路径：

```text
进入首页看板
→ 发现未分类或异常分类
→ 进入流水页修正
→ 必要时进入规则页新增规则
→ 返回首页看结果变化
```

预算关注路径：

```text
首页预算提醒
→ 进入预算页查看详情或调整预算
→ 返回首页继续观察
```

### 7.9 页面状态设计
每个页面都应覆盖以下状态类型：

1. 初始加载状态
2. 空状态
3. 正常数据状态
4. 错误状态
5. 操作成功反馈状态

### 7.10 前端状态管理建议
前端状态建议分成三类：

1. 页面本地状态
2. 查询条件状态
3. 服务端数据状态

设计原则：

1. 服务端数据以接口结果为准
2. 查询条件可在页面间有限共享
3. 不把复杂业务真相缓存到前端自行维护

### 7.11 前端交互设计原则
V1 前端原则：

1. 先展示价值，再要求配置
2. 所有关键结果可回到明细
3. 所有智能结果都允许人工纠偏
4. 页面跳转围绕任务，不围绕功能树
5. 状态反馈必须明确

### 7.12 本部分结论
V1 的页面方案应以首页承载结果、导入页承载入口、流水页承载纠偏、规则页承载增强、预算页承载约束，形成一条清晰、低门槛、可回溯的用户操作路径。

## 八、技术选型与实现建议

### 8.1 选型目标
V1 的技术选型应满足以下原则：

1. 快速落地
2. 结构清晰
3. 开发友好
4. 可扩展

V1 适合的是“成熟工程栈”，而不是“实验性栈”或“过于重型的企业栈”。

### 8.2 前端技术选型建议
推荐方向：

1. 框架：React
2. 构建工具：Vite
3. 路由：React Router
4. UI 组件基础：Ant Design 或相近成熟组件库
5. 图表库：ECharts 或 Recharts
6. HTTP 请求：Axios 或基于 `fetch` 的统一封装
7. 状态管理：优先轻量方案

为什么这样选：

1. React + Vite 适合快速搭建前端应用
2. 组件生态成熟，适合后台型与数据型页面
3. 图表库成熟，足以支撑首页看板
4. 对文件上传、表格、筛选、表单等场景支持较好

前端不建议：

1. 一开始就引入复杂微前端体系
2. 一开始就重度依赖全局状态管理
3. 为了 V1 过早引入 SSR 或同构复杂度

### 8.3 后端技术选型建议
推荐方向：

1. 语言：Java
2. 框架：Spring Boot
3. ORM / 数据访问：MyBatis-Plus、JPA，或更偏 SQL 控制的持久层方案
4. 文件解析：CSV 解析库 + Excel 解析库
5. 参数校验：Bean Validation
6. API 风格：RESTful JSON API

为什么这样选：

1. 当前仓库已有 Java 风格 `.gitignore` 痕迹
2. Spring Boot 适合单体后端分层架构
3. 对文件上传、数据库操作、接口组织、异常处理支持成熟
4. 适合后续扩展为更完整的业务系统

后端不建议：

1. V1 上来就拆微服务
2. 上来就做复杂异步任务中台
3. 过度引入消息队列、缓存中间件和分布式事务体系

### 8.4 数据库选型建议
V1 数据存储建议使用关系型数据库。

推荐方向：

1. MySQL
2. PostgreSQL

关系型数据库适合本项目的原因：

1. 核心数据结构明确
2. 交易、规则、预算、导入任务都适合结构化建模
3. 查询场景以筛选、聚合、统计为主
4. 后续做索引和统计优化路径清晰

### 8.5 文件存储与上传处理建议
建议方案：

1. 文件上传后，先保存原始文件引用或临时文件
2. 后端解析完成后，以数据库保存结构化交易数据
3. 文件本身可根据实现复杂度决定是否长期保留

V1 推荐做法：

1. 上传文件先走本地文件存储或服务器目录临时保存
2. 解析完成后以结构化数据为主
3. 原文件是否长期保存作为可选能力

### 8.6 图表技术选型建议
推荐方向：

1. ECharts
2. Recharts

从当前需求看，更推荐 `ECharts`，原因是：

1. 分类分布、趋势线、预算对比等表达较成熟
2. 数据展示形式更丰富
3. 后续扩展图表时弹性更大

### 8.7 前端工程结构建议
建议结构方向：

1. `pages`
2. `components`
3. `features`
4. `services`
5. `types`
6. `utils`

这样组织的好处：

1. 页面与业务模块关系明确
2. 不容易形成“所有代码都堆在 components”
3. 适合后续按模块持续扩展

### 8.8 后端工程结构建议
建议结构方向：

1. `controller`
2. `service`
3. `repository` / `mapper`
4. `domain` / `entity`
5. `dto` / `vo`
6. `module`
7. `common`

设计重点：

1. 包结构服务于领域边界，而不是纯技术堆叠
2. 控制器层保持薄
3. 核心业务逻辑集中在服务层或领域模块中

### 8.9 状态管理与数据获取建议
建议思路：

1. 页面本地交互状态放在页面内部
2. 服务端数据通过统一查询机制管理
3. 页面筛选条件按页面或局部共享处理
4. 跨页面的业务真相全部以后端返回为准

如果需要更明确的方向，建议采用：

1. React Query / TanStack Query 一类服务端状态管理方案
2. 普通本地状态处理页面交互
3. 仅在确实出现大量跨页面共享状态时，再考虑更重全局状态工具

### 8.10 安全与权限实现建议
建议范围：

1. 文件上传类型和大小做基本校验
2. 接口输入做参数校验
3. 统一异常处理
4. 敏感信息最小展示
5. 后续若需要登录，再追加认证机制

### 8.11 性能与扩展实现建议
建议：

1. 交易列表查询支持分页
2. 常用筛选字段预留索引思路
3. 看板统计按时间范围聚合时避免全表无约束扫描
4. 预算计算复用交易与分类结果，不重复实现一套新逻辑

暂不建议：

1. 上缓存中间件
2. 上异步流处理平台
3. 上复杂预计算体系

### 8.12 开发与协作工具建议
建议工程层面配套这些基础能力：

1. 统一代码风格工具
2. 基础单元测试与接口测试
3. 环境配置区分开发环境与生产环境
4. API 文档工具或最少的接口约定文档
5. 基础日志输出能力

如果采用推荐栈：

1. 前端可配 ESLint + Prettier
2. 后端可配统一格式化与静态检查
3. 接口可通过 Swagger / OpenAPI 或同类工具维护

### 8.13 本部分结论
V1 最适合采用 React + Vite 的前端栈、Spring Boot 单体后端、关系型数据库持久化和轻量文件解析方案，以成熟稳健替代过度前沿或过度复杂的技术组合。

## 九、非功能设计与工程约束

### 9.1 设计目标
V1 的非功能设计不追求企业级复杂能力，但必须保证以下几点：

1. 系统可用
2. 数据可信
3. 工程可持续
4. 范围受控

核心原则：

`V1 不求重，但必须稳。`

### 9.2 可用性设计约束
必须满足的可用性要求：

1. 用户能够完成从导入到账单查看的完整流程
2. 用户能够完成从自动分类到人工修正的纠偏流程
3. 用户能够完成预算设置并看到预算状态
4. 首页、流水页、预算页之间的跳转逻辑清晰
5. 页面不会因为缺少配置而完全不可用

设计约束：

1. 没有预算时，首页仍可展示消费分析
2. 没有规则时，系统仍可执行默认分类
3. 有未分类交易时，系统仍可继续展示看板
4. 某次导入部分失败时，成功记录仍应可进入系统

### 9.3 性能设计约束
建议性能目标：

1. 页面基础加载应足够轻量
2. 导入后的处理应在用户可接受的时间内完成
3. 看板查询应优先返回核心结果
4. 流水筛选和分类修改后应有及时响应

工程约束：

1. 交易列表必须支持分页
2. 看板统计必须基于时间范围查询
3. 不允许无筛选条件的大范围全量明细拉取成为默认页面行为
4. 预算计算应直接复用已有交易与分类结果

### 9.4 数据准确性与一致性约束
必须满足的准确性要求：

1. 同一笔交易在流水页、看板、预算中的金额口径一致
2. 手动修正分类后，预算和看板结果同步更新
3. 未分类交易不会被错误计入某个消费分类
4. 重复导入不会导致同一交易被无限重复统计
5. 总收入、总支出、净结余的计算口径稳定一致

工程约束：

1. 交易事实层必须是唯一可信来源
2. 最终分类结果必须是唯一可消费分类结果
3. 看板与预算不得各自实现不同分类口径
4. 前端不得自行补算与后端冲突的关键业务结果

### 9.5 可维护性设计约束
必须满足的可维护性原则：

1. 前后端职责清晰
2. 页面职责清晰
3. 领域模块边界清晰
4. 数据模型语义清晰
5. 错误处理方式统一

工程约束：

1. 不允许在前端页面中堆叠过多业务判断
2. 不允许在后端 Controller 层沉积大量核心业务逻辑
3. 不允许导入、分类、预算、看板逻辑相互复制
4. 通用错误响应、状态码和结果结构必须统一

### 9.6 可扩展性设计约束
需要预留扩展的位置：

1. 新账单模板适配
2. 更丰富的规则能力
3. 更复杂的预算周期
4. 更丰富的看板维度
5. 用户体系接入

但必须控制的边界是：

1. 只预留结构，不提前实现复杂能力
2. 不为了未来功能引入当前无必要的基础设施
3. 不在 V1 阶段引入过多抽象层

### 9.7 错误处理与可恢复性约束
错误类型划分：

1. 用户输入错误
2. 业务状态型异常
3. 系统运行异常

设计约束：

1. 用户输入错误要明确提示并可立即修复
2. 业务状态型异常要转成正常页面状态
3. 系统运行异常要统一日志记录并返回通用错误响应
4. 不能把“未分类”这种正常业务状态处理成系统报错

恢复性要求：

1. 导入失败后用户可重新上传
2. 分类修改失败后用户可重新提交
3. 预算保存失败后用户可继续编辑重试
4. 某些数据为空时，页面应可继续使用其他能力

### 9.8 安全与隐私约束
基础要求：

1. 文件上传做类型和大小限制
2. 后端对所有输入参数做校验
3. 页面展示遵循最小必要原则
4. 日志中避免无意义输出完整敏感账单内容
5. 敏感数据处理以产品目标所需为限

### 9.9 测试设计约束
建议测试层次：

1. 后端单元测试
2. 后端接口测试
3. 前端页面级联调验证
4. 关键业务场景验证

设计约束：

1. 统计口径类逻辑必须有测试保障
2. 分类规则命中逻辑必须有测试保障
3. 预算状态判断必须有测试保障
4. 文件解析至少要有基础样例验证

### 9.10 工程边界控制
V1 必须做：

1. 导入
2. 标准化交易
3. 自动分类
4. 手动修正
5. 看板统计
6. 预算配置与预警

V1 不应顺手扩出去做：

1. 登录与用户体系
2. 云同步和多端同步
3. 多人协作账本
4. 理财和借贷模型
5. 高级规则引擎
6. 复杂消息通知系统

### 9.11 验收与交付约束
必须可验证的结果：

1. 用户成功上传账单并产生交易记录
2. 自动分类结果能够形成初步消费结构
3. 未分类交易可被定位与修正
4. 首页可展示核心指标和预算状态
5. 预算设置后可正确显示正常 / 接近预算 / 已超支状态
6. 页面跳转和主流程完整可走通

### 9.12 本部分结论
V1 的非功能设计应以可用性、准确性、一致性和可维护性为核心，以统一状态处理和清晰工程边界为约束，保证系统在有限范围内稳定可交付。

## 十、实施阶段建议与方案结论

### 10.1 实施目标
这一部分不是直接写开发任务清单，而是从方案层面明确：

1. V1 应按什么顺序实施最稳
2. 哪些模块必须先落地
3. 哪些模块可以后置
4. 如何保证开发过程始终围绕主链路推进

核心原则：

`先打通主链路，再补强体验和扩展能力。`

### 10.2 推荐实施顺序
基于当前方案，V1 最合理的实施顺序建议为：

1. 交易事实层先行
2. 导入链路打通
3. 自动分类能力接入
4. 看板聚合输出落地
5. 预算能力接入
6. 规则管理和体验增强补齐

### 10.3 分阶段实施建议
V1 可以分成 4 个实现阶段。

阶段一：数据基础层

应完成：

1. 后端项目基础结构
2. 数据库基础模型
3. 交易实体与导入任务实体
4. 基础交易查询接口
5. 前端基础工程和页面骨架

阶段二：导入与分类主链路

应完成：

1. 文件上传接口
2. 模板识别和解析器框架
3. 标准化交易转换
4. 自动分类引擎
5. 流水页明细展示
6. 单笔交易分类修改

阶段三：看板与预算能力

应完成：

1. 首页看板聚合接口
2. 总览指标与分类分布
3. 趋势统计
4. 预算配置接口
5. 预算计算与预警逻辑
6. 首页预算展示与预算页

阶段四：规则管理与体验完善

应完成：

1. 规则管理页
2. 用户规则创建/删除
3. 未分类快捷处理优化
4. 空状态和错误状态完善
5. 页面跳转与交互细节优化

### 10.4 模块优先级收敛
P0 核心交付：

1. 前后端基础工程
2. 数据库与核心数据模型
3. 账单导入与标准化
4. 自动分类
5. 流水页展示
6. 分类人工修正
7. 首页核心看板
8. 月总预算与分类预算
9. 预算状态与预警

P1 增强交付：

1. 用户规则管理
2. 未分类快捷处理优化
3. 更完整的错误反馈
4. 首页联动跳转优化
5. 导入结果展示优化

P2 后续迭代：

1. 更多账单模板支持
2. 更复杂规则能力
3. 更丰富图表维度
4. 更灵活预算体系
5. 登录与云端能力

### 10.5 关键实施依赖关系
明确依赖如下：

1. 看板依赖交易事实层和最终分类结果
2. 预算依赖交易事实层和最终分类结果
3. 规则管理依赖分类引擎存在
4. 分类修改依赖交易查询能力存在
5. 导入结果展示依赖导入任务模型存在

### 10.6 风险点与规避建议
风险一：导入模板适配低估

规避建议：

1. V1 先聚焦少量高价值模板
2. 先搭建“解析器框架”，再逐步扩模板
3. 不要一开始承诺全平台全格式兼容

风险二：分类逻辑失控

规避建议：

1. V1 只做简单规则匹配
2. 明确人工修正优先
3. 复杂规则能力留待后续迭代

风险三：统计口径不一致

规避建议：

1. 后端统一口径
2. 以最终分类结果为唯一统计依据
3. 前端不自行计算关键业务结果

风险四：页面先行导致结构反噬

规避建议：

1. 先稳住交易、分类、预算三个核心模型
2. 页面围绕已有服务能力建设
3. 看板接口用聚合模式，避免前端拼装过重

风险五：V1 范围膨胀

规避建议：

1. 以 PRD 主链路为唯一边界
2. 新增内容必须明确判断是否阻塞 V1 成立
3. 非阻塞项全部后置

### 10.7 实施完成标准
从方案落地角度，V1 实施完成至少意味着：

1. 用户可导入账单并得到结构化交易
2. 系统可自动分类并允许人工修正
3. 首页可展示消费结构与预算状态
4. 用户可设置预算并获得预警
5. 主要页面可形成闭环跳转
6. 数据结果在流水、看板、预算之间保持一致

### 10.8 方案总结
整份方案文档最终设计结论：

1. V1 是一个前后端分离的单用户 Web 财务分析系统
2. 系统以标准化交易记录作为唯一事实核心
3. 自动分类是连接导入与分析的关键桥梁
4. 首页看板和预算能力是面向用户的核心输出
5. 人工修正和规则管理是建立长期可用性的关键增强路径
6. 技术实现上应采用成熟、轻量、可扩展的工程架构，而不是过度设计

### 10.9 一句话结论
这份方案设计文档的最终结论可以概括为：

`Smart Finance Dashboard V1 应被实现为一个以交易事实为核心、以分类规则为桥梁、以看板分析和预算预警为输出的轻量型前后端分离财务看板系统。`

这份方案文档完成后，最自然的下一步是进入：

2. 数据库表结构设计
